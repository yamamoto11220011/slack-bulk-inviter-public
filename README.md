# Slack Bulk Inviter

Slack のチャンネルに大量のユーザーを一括招待するためのデスクトップアプリです。
GUI (Electron) と CLI の両方で利用できます。

## このツールについて

大規模な Slack ワークスペースなど、多数のユーザーが所属する環境で、識別子パターンに基づいてユーザーを絞り込み、特定のチャンネルにまとめて招待する用途を想定しています。

Slack の Bot/API トークンの発行権限がなくても、ブラウザでログインした際のセッション情報を利用して Slack Web API を呼び出す仕組みです。

## 利用上の注意

- **本ツールは Slack の公式な API 利用方法ではありません。** ブラウザセッションから取得した `xoxc` / `xoxd` トークンを使用しており、これは Slack が公式にサポートする認証方式（OAuth による Bot/User トークン）とは異なります。
- Slack の利用規約上、「未公開 API の使用」や「アクセス制御の回避」に該当する可能性があります。Slack がトークンの無効化やアカウント停止等の措置を取る権利を持っている点をご理解ください。
- 実際には自分のアカウントで自分が手動でもできる操作を自動化しているに過ぎず、同様のツールが多数公開されていますが、**利用は自己責任** でお願いします。
- Slack 側の仕様変更により予告なく動作しなくなる可能性があります。
- 他人のワークスペースへの不正アクセスや、迷惑行為（スパム招待等）には絶対に使用しないでください。
- 管理者に Bot/API トークンの発行を依頼できる場合は、そちらの正規ルートを強く推奨します。本ツールはそれが難しい場合の代替手段です。

## 個人情報の取り扱いについて

本ツールは Slack ワークスペースのユーザー情報（表示名、氏名、アバター画像等）をローカルに保存します。これらは個人情報に該当する場合があります。

- **ローカルデータは AES-256-GCM で暗号化** して保存されますが、端末自体のセキュリティ（ログインパスワード、ディスク暗号化等）も適切に管理してください。
- ユーザー情報のスクリーンショットや画面共有には十分注意してください。学籍番号等の識別子からメールアドレスが推測できる場合があります。
- 本ツールで取得したユーザーデータを第三者に共有したり、本来の目的（チャンネル招待）以外に流用しないでください。
- 不要になったデータは速やかに削除してください。アプリのデータディレクトリは以下にあります:
  - macOS: `~/Library/Application Support/slack-bulk-inviter/`
  - Windows: `%APPDATA%/slack-bulk-inviter/`
  - CLI: `~/.slack-bulk-inviter/`

## 機能

- **ユーザー一覧**: Slack ワークスペースのユーザーを取得し、識別子パターンで自動分類
- **カテゴリフィルタ**: コホート・所属グループごとにユーザーを絞り込み、一括選択
- **識別子フィルタ**: GUI 上で「識別子あり / なし」でユーザーを即座に絞り込み
- **GUI ソート**: 学籍番号順、逆順、表示名順、カテゴリ順で一覧を並び替え
- **複数チャンネル一括招待**: 選択したユーザーを複数チャンネルへ順番に招待
- **停止ボタン**: 実行中の一括招待を途中で停止
- **ローカルDB**: SQLite にユーザー、チャンネル、所属、コメント履歴を保存（既存の暗号化DBから移行対応）
- **ダークモード**: OS の設定に自動追従

## 開発者としてやったこと

このプロダクトでは、単なる一括送信スクリプトではなく、実運用に耐えるローカル SaaS ライクなツールとして設計しました。特に重視したのは、`対象抽出`、`誤送信防止`、`負荷制御`、`ローカル保存` を1つの体験にまとめることです。

- **Electron アプリとして設計**: `main`、`preload`、`renderer` を分離しつつ、Slack 通信・同期・招待・DM・ブロードキャストの業務ロジックは `src/core` に集約しました。
- **Slack のブラウザセッション再利用**: OAuth/Bot 権限がない環境でも使えるよう、ログイン済みセッションから認証情報を扱う構成にし、保存時は OS の保護機能や暗号化を使ってローカル管理しています。
- **セグメント抽出と安全設計**: カテゴリ一括選択、学籍番号なしユーザーの自動除外、対象ユーザーの可視化、送信前確認を実装し、先生や対象外への誤送信を避けやすくしました。
- **同期の高速化**: 全チャンネルの重い所属同期を毎回やらず、通常同期はチャンネル一覧中心にして、必要なメンバー照会だけを対象チャンネル単位で走らせるように最適化しました。
- **送信基盤の拡張**: 一括招待だけでなく、個別 DM、繰り返しブロードキャスト、予約送信、画像・動画・ローカルファイル添付に対応できるようにしました。
- **発熱と負荷の抑制**: 並列数、再試行、送信間隔、進捗通知頻度を調整し、添付ファイル送信もストリームベースにして、PC が重くなりにくいようにしています。
- **フロントエンドの操作性改善**: 仮想化テーブル、遅延ロード、タブごとの独立下書き保存、中央配置の DM 操作パネルなどを入れて、人数が多くても扱いやすい UI にしました。

## 技術的な見どころ

- **ローカル完結型のアーキテクチャ**: ユーザーデータや送信設定をローカルに持ち、外部サーバーを前提にせず完結する構成です。
- **安全性を先に作る設計**: 配信機能より先に、除外条件、可視化、確認フロー、停止機能を揃えることで、実利用時の事故コストを下げています。
- **機能追加しやすいデータモデル**: `users`、`channels`、`channel_memberships`、`messages`、`broadcast_tasks` を分けて持つことで、招待だけでなく監査や配信系にも拡張しやすくしています。

## GUI で追加したこと

- 左サイドバーに「学籍番号あり」「学籍番号なし」のフィルタを追加しました。
- ユーザー一覧に Slack の `name` をそのまま表示し、学籍番号として扱う値を確認しやすくしました。
- ユーザー一覧の並び順を GUI から切り替えられるようにしました。
- 招待先チャンネルを複数選択できるようにしました。
- 一括招待の進捗表示に加えて、途中で停止できるようにしました。

識別子の有無は、現在は Slack の `name` が「4文字以上の英数字ベースで、数字を少なくとも1つ含む」場合に `あり` と判定しています。組織ごとの厳密な命名規則がある場合は、そのルールに合わせて判定条件を調整してください。

## データ同期の考え方

- `ユーザー`: Slack のユーザー一覧を同期します。
- `チャンネル`: チャンネル一覧を同期します。必要な所属メンバーの照会は対象チャンネルごとに行います。

ローカルDBは `SQLite` を使っており、`users`、`channels`、`channel_memberships`、`messages` を分けて保持します。複数チャンネルへの一括招待や所属情報の管理を、あとから拡張しやすい構成にしています。

## macOS で初めて起動する場合

コード署名されていないアプリのため、macOS の Gatekeeper によりブロックされます。
以下のいずれかの方法で起動してください。

### 方法 1: 右クリックから開く

1. Finder で `Slack Bulk Inviter.app` を探す
2. **右クリック（または Control + クリック）** → **「開く」** を選択
3. 警告ダイアログが表示されるので **「開く」** をクリック

2回目以降はダブルクリックで普通に起動できます。

### 方法 2: ターミナルから起動

```bash
xattr -cr "dist/mac-arm64/Slack Bulk Inviter.app"
open "dist/mac-arm64/Slack Bulk Inviter.app"
```

`xattr -cr` で検疫属性を除去すると、以降はダブルクリックで開けます。

## CLI の使い方

```bash
# ログイン（ブラウザが開きます）
npm run cli -- auth login

# 認証状態の確認
npm run cli -- auth status

# ユーザー・チャンネルデータの同期
npm run cli -- sync all

# ユーザー一覧（カテゴリでフィルタ）
npm run cli -- users list --category 2025-spring

# チャンネル一覧（参加中のみ）
npm run cli -- channels list --member

# 招待のプレビュー（実行しない）
npm run cli -- invite --channel general --category 2025-spring --dry-run

# 招待を実行
npm run cli -- invite --channel general --category 2025-spring
```

## ユーザー分類の設定

`config/categories.yml` は公開用のサンプル設定です。
Slack の `name`（ユーザー名）フィールドに対する前方一致でマッチします。

```yaml
categories:
  - id: "2025-spring"
    label: "2025 Spring Intake"
    type: "cohort"
    patterns:
      - "25sp"
      - "25su"
```

実運用では、以下の優先順でローカル専用設定を読み込みます。

- GUI アプリ: `~/Library/Application Support/slack-bulk-inviter/categories.local.yml`
- 開発時: `config/categories.local.yml`
- CLI: `~/.slack-bulk-inviter/categories.local.yml`

`config/categories.local.yml` は `.gitignore` に入っているので、学校名や学籍番号プレフィックスのような運用情報はそこへ置くのがおすすめです。

## GitHub に公開する時の注意

- `config/categories.yml` はサンプルのままにして、実運用ルールは `categories.local.yml` に置いてください。
- `app.log`、`*.sqlite`、`*.enc`、`*.log` は Git に載らないよう `.gitignore` で除外しています。
- Electron/CLI の認証情報はリポジトリ外の OS 管理ディレクトリに保存されるため、通常は一緒に公開されません。
- ただし、**既存の git 履歴に含まれる作者メールアドレスや過去設定は別問題** です。公開前に新しい public リポジトリへ初回コミットし直すか、履歴を書き換えてから push してください。
